## 1. System Overview

NutriAI is a Streamlit application that generates a personalized 7-day, 3-meal-per-day meal calendar for students and shared households with clinical, allergy, diet, budget, and meal-prep constraints. The app starts with an empty household so users can add or remove people themselves. NutriAI combines each person's filters so the final calendar is safe for everyone, while RDA math is anchored to a selected nutrition target member. The main workflow is:

`Household intake -> filter aggregation -> validation -> clinical/allergy hard filters -> candidate generation -> Bloom precheck -> embedding-aware ranking -> diversity assignment -> 1-4 week visual planner -> macro/micro analysis -> grocery export + restaurant mode`

The app is intentionally conservative. Clinical restrictions and allergens are hard filters. Ranking is only applied after a recipe has passed safety checks. Calendar swap/replacement controls only use meals that have already passed the household safety screen. Restaurant Mode is labeled as a risk estimate because independent restaurants do not always publish ingredient-level allergen data.

## 2. Data Pipeline

The offline food snapshot is generated by `code/scripts/build_food_snapshot.py` from official USDA FoodData Central CSV archives:

- Foundation Foods, April 2026
- SR Legacy, April 2018
- Survey/FNDDS, October 2024

The pipeline downloads the ZIP archives, extracts `food.csv`, `food_nutrient.csv`, and nutrient metadata, maps both FDC nutrient IDs and legacy FNDDS nutrient numbers, pivots nutrient rows into structured food records, deduplicates normalized food descriptions, and adds keyword-based metadata for allergens and clinical caution flags.

Final local snapshot: **12,000 structured USDA-derived records** in `data/food_snapshot.csv`.

The meal planner uses `data/recipe_catalog.csv`, a curated catalog of 99 recipe candidates with complete macro/micronutrients, clinical flags, diet compatibility, allergens, cross-contact risks, GI estimates, sodium, potassium, and grocery fields. This separation keeps the nutrition data pipeline real while making the actual planner safe and demoable.

## 3. Core Capabilities

NutriAI supports all six required capabilities:

1. **Clinical filtering:** IBS, GERD, Type 2 Diabetes, and Hypertension are enforced with low-FODMAP, GERD trigger, low-GI, sodium, and DASH-style rules.
2. **Allergy exclusion:** Direct allergens and strict cross-contact risks are excluded before ranking.
3. **Dietary handling:** Vegan, vegetarian, pescatarian, non-vegetarian, mixed meal diets, no pork, no beef, and halal-style restrictions are supported.
4. **Diversity engine:** The assignment step avoids repeated recipes and reports a diversity score.
5. **Nutrient analytics:** Per-meal and daily totals are computed for calories, protein, carbs, fat, fiber, iron, calcium, B12, vitamin D, zinc, sodium, potassium, and magnesium. Daily values are compared to age/sex RDA or AI targets.
6. **Sub-60-second generation:** The app logs and displays generation time.

## 4. BAX-423 Techniques and Benchmarks

**Technique 1: Bloom-filter screening.**  
Each recipe stores precomputed Bloom bits for allergen and unsafe-condition tokens. At runtime the planner uses this as a fast precheck before exact rule verification. Benchmark on an expanded recipe catalog:

- Exact token scan: 2305.35 ms
- Bloom precheck: 1710.98 ms
- Impact: 1.35x faster

**Technique 2: Hash embeddings + multi-stage ranking.**  
User preferences, diet, allergies, clinical conditions, and micronutrient priorities are embedded with a lightweight hashing vector. Candidate recipes are scored by embedding similarity, calorie fit, protein/fiber fit, sodium fit, condition-specific boosts, and category diversity.

Benchmark against a calorie-only baseline:

- Calorie-only baseline: 0 persona-specific review flags; average diversity 0.82
- Optimized ranker: 0 persona-specific review flags; average diversity 0.93
- Impact: stronger diversity while preserving hard-filter safety.

Generation benchmark:

- Requirement: under 60 seconds
- Measured locally: mean 0.092 seconds, max 0.108 seconds across required personas

## 6. Student Extensions

**Household Planner Mode:** Shows the generated plan in polished week and day views instead of a table. Users can add/remove household members, see color-coded allergy chips, safely swap or replace meals, and choose a 1-4 week look-ahead horizon while preserving clinical/allergy filters. The first week remains the required no-repeat rubric plan; later weeks avoid repeats until the safe candidate pool is exhausted.

**Visual Design:** Uses a restrained light green dashboard theme with compact cards, metrics, and charts so the app feels like a finished student meal-planning product rather than a table prototype.

**Interactive Nutrition Charts:** Adds a macro split donut chart and selectable daily trend chart for calories, protein, fiber, sodium, iron, calcium, and B12.

**Grocery List Mode:** Aggregates all meals in the selected planning horizon into a categorized shopping list, tracks reuse frequency, gives quantity hints, estimates plan cost, scales estimates by household size, and exports to CSV.

**Restaurant Mode:** Provides a student-oriented safe/caution/avoid menu ranking over a representative San Francisco neighborhood restaurant snapshot. It applies the same clinical, allergy, diet, and sodium/GI rules but labels results as risk estimates rather than guarantees.

## 7. Limitations and Next Steps

The app is a class project, not medical advice. GI values, clinical flags, and restaurant allergen confidence are estimated from curated metadata. Restaurant Mode uses representative SF-style menu items, not live restaurant menus, and should be validated against official restaurant allergen PDFs or live APIs before real-world use. Future work would add real restaurant API ingestion, barcode scanning, a larger recipe database, adaptive feedback learning, and dietitian-reviewed clinical rules.
